## What This Repo Does

A TypeScript VSCode extension (publisher: `TabbyNotes`) that provides editor support for
Writerly files. It activates automatically for files with the `.wly` extension.

Features:
- Syntax highlighting (TextMate grammar)
- Indentation and tag validation (real-time diagnostics)
- Handle system: `F12` go-to-definition, `F2` rename, unused-handle warnings
- Hover preview for images and files referenced by `src=` / `original=` paths
- `Cmd+Shift+O` (Mac) / `Ctrl+Shift+O` opens the file under cursor with the system default app
- Path autocomplete for `src=` and `original=` attribute values
- Writerly Files tree view in the VSCode activity bar
- `writerly.restart` command resets all providers

## Build — IMPORTANT

**After any successful implementation, you must compile:**

```sh
npm run compile
```

This transpiles `src/**/*.ts` → `out/**/*.js` (the compiled output VSCode actually loads).
The `out/` directory is what ships in the extension package — edits to `src/` have no effect
until `npm run compile` is run.

To watch for changes during development:
```sh
npm run watch
```

To package the extension:
```sh
npm run vscode:prepublish   # compiles, then packages
```

## Architecture

### Entry Point

`src/extension.ts` calls `new WriterlyController(context)` on activation. The controller
instantiates all providers in order and registers the `writerly.restart` command, which calls
`reset()` on each provider.

### Provider Reset Pattern

Every provider exposes a `reset()` method:
- **Stateless providers** (`WriterlyFileOpener`, `WriterlyHoverProvider`) implement a no-op `reset()`.
- **Stateful providers** (`WriterlyLinkProvider`, `WriterlyCompletionProvider`,
  `WriterlyFileProvider`) clear internal caches and re-index the workspace.

### `WriterlyDocumentWalker` — the Core Parser

All providers that need to understand `.wly` structure call through `WriterlyDocumentWalker`.
It implements a line-by-line state machine with three zones:

| Zone | Meaning |
|---|---|
| `Zone.Text` | Default; blank lines, text paragraphs, `\|\>` tag lines |
| `Zone.Attribute` | Lines immediately after a `\|>` tag (key=value pairs) |
| `Zone.CodeBlock` | Inside a ` ``` ` fence |

`State` carries `zone`, `maxIndent`, `minIndent`, `codeBlockStartIndent`,
`codeBlockStartLineNumber`.

**Two usage modes:**

1. `WriterlyDocumentWalker.walk(document, callback)` — full document traversal; the callback
   receives `(stateBeforeLine, lineType, stateAfterLine, lineNumber, indent, content)` for every
   line. Used by `WriterlyLinkProvider` to extract handle definitions and usages.

2. `WriterlyDocumentWalker.onTheFlyLineClassification(document, position)` — classifies only
   the line at `position` by walking only the minimal context above it. Used by
   `WriterlyCompletionProvider` to decide whether to offer completions.

**`LineType` enum:**

| LineType | When |
|---|---|
| `Tag` | Line starting with `\|>` |
| `Attribute` | `key=value` line in attribute zone |
| `AttributeZoneComment` | `!!` line in attribute zone |
| `Text` | Text paragraph line |
| `TextZoneComment` | `!!` line in text zone |
| `TextZoneEmptyLine` | Blank line |
| `CodeBlockOpening` | ` ``` ` opening fence |
| `CodeBlockLine` | Line inside a code block |
| `CodeBlockClosing` | ` ``` ` closing fence |

### `WriterlyStaticValidator` — Syntax Diagnostics

Called from inside `WriterlyLinkProvider.walkDocument` for every line. Produces
`vscode.Diagnostic` errors for:

| Code | Condition |
|---|---|
| d1 | Indentation too large |
| d2 | Indentation not a multiple of 4 |
| d3 | Indentation too low |
| d4 | Code block opening inside a code block |
| d5 / d6 | Unclosed code block (at opening line and at EOF) |
| d7 | Empty tag (`\|>` with nothing after it) |
| d8 | Invalid tag name (must match `[a-zA-Z_:][a-zA-Z0-9._:-]*`) |
| d9 | Spaces in code block info annotation (e.g. ` ```python foo ` is invalid) |
| d10 | Tabs in initial whitespace |

### `WriterlyLinkProvider` — Handle System

The most complex provider. It maintains workspace-wide state and implements:
- `vscode.DocumentLinkProvider` — underlines `>>handle` usages
- `vscode.DefinitionProvider` — `F12` jumps to the `handle=` definition
- `vscode.RenameProvider` — `F2` renames a handle everywhere in the workspace
- `vscode.CodeActionProvider` — quick fixes for undefined handles
- `vscode.CompletionItemProvider` — completes `>>` handle names

**Handle syntax:**
- **Definition**: `handle=<name>` attribute line (in an attribute zone after any `|>` tag)
- **Usage**: `>><name>` anywhere outside of code blocks and tag lines

**Handle name character set** (Unicode-aware):
- Allowed interior: letters (`\p{L}`), digits (`\p{N}`), marks (`\p{M}`), `_`, `.`, `:`, `-`, `^`
- Must end with: letter, digit, mark, `_`, or `^` (`\p{L}\p{N}\p{M}_^`)

**Internal state:**
- `definitions: Map<HandleName, HandleDefinition[]>` — maps each handle name to all files and
  ranges where it is defined (multiple definitions are allowed)
- `documentLinks: Map<FSPath, vscode.DocumentLink[]>` — cached usage links per file
- `usageCounts: Map<HandleName, number>` — total usage count across all files (used for
  unused-handle warnings)
- `documentRoots: FSPath[]` — list of assemblable root directories. A directory
  is an extension document root when it contains at least one direct `.wly` file.
  Broad document-tree membership ignores `#` path segments. Handle lookup and
  completion use the nearest accessible hash island with a single non-ambiguous
  definition, duplicate diagnostics compare only definitions in the same hash
  island, and rename applies to the whole document tree.

**Lifecycle:** on startup, `initializeAsync()` discovers assemblable root directories
and processes all Writerly files in the workspace. A `FileSystemWatcher` keeps state current as files are
created, changed, deleted, or renamed. Changes are debounced via `triggerTreeRevalidation`.

Unused handle warnings are controlled by the `writerly.enableUnusedHandleWarnings` setting.

### `WriterlyHoverProvider` — Image & File Hover

Stateless. On hover over any token that `WriterlyFileOpener.getResolvedFilePathAtPosition`
resolves to an existing file, it shows:
- Action links: "Open with default", "Open with VSCode" / "Open as image", "Show in Explorer"
- Absolute file path
- Inline image preview (for PNG, JPG, SVG, etc.)
- File size in KB or MB

### `WriterlyCompletionProvider` — Path Autocomplete

Triggered on `=`, ` `, `/` in Writerly files. Offers completions only when the cursor is in
`src=<path>` or `original=<path>` position. Builds an in-memory file tree of all image files
in the workspace (PNG, JPG, JPEG, GIF, SVG, WebP, BMP, IPE, PSD, TIFF — both cases).
Directory completions re-trigger suggest automatically. File completions show an inline image
preview in the documentation popup.

### `WriterlyFileOpener` — Open Under Cursor

Stateless. Resolves the file path token under the cursor (stops at whitespace and forbidden
characters `[\s'"=\[\]\{\}\(\);!<>|]`) relative to the workspace root, then opens it via:
- `OpeningMethod.WITH_DEFAULT` — system default app (`open` on macOS, etc.)
- `OpeningMethod.WITH_VSCODE` — VSCode text editor
- `OpeningMethod.AS_IMAGE_WITH_VSCODE` — VSCode image viewer

### `WriterlyFileProvider` — Activity Bar Tree View

Displays all Writerly files in the workspace in a tree, mirroring the directory structure.
Auto-reveals the currently active editor's file in the tree. Refreshes on file system changes.

## Language Configuration

`writerly.configuration.json` configures VSCode's built-in language behaviors:
- **Line comment**: `!!`
- **Auto-indent after tag**: pressing Enter after a `|>` line auto-indents by 4 spaces
  (`onEnterRules`)
- **Bracket pairs** and surrounding pairs for `{}`, `[]`, `()`, `""`
- **Offside folding** (indentation-based code folding)

## Syntax Highlighting

`syntaxes/writerly.json` is a TextMate grammar that highlights:
- `|> TagName` — tag lines
- `key=value` — attribute lines
- `!! comment` — comment lines
- ` ``` ` fences — code blocks
- `>>handle` — handle usages

If the grammar needs updating (e.g. new syntax forms), edit `syntaxes/writerly.json` and then
run `npm run compile` (the grammar itself does not require compilation, but compile anyway to
keep the build consistent).

|---|---|---|
